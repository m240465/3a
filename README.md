# GLH Task 3A — Full Feedback Documentation Pack

## Document 1 — Feedback Planning Document

### 1.1 Purpose of collecting feedback

The purpose of collecting feedback is to evaluate how effective the GLH prototype is from both a user experience perspective and a technical development perspective. The prototype includes customer-facing features such as the homepage, supplier browsing, product viewing, checkout, basket quantity controls, login/register, and accessibility settings. It also includes supplier-side features such as the dashboard, supplier detail editing, product adding, product updating, product visibility controls, and the sales summary.

Feedback is important because a system can appear to work correctly from the developer’s point of view but still be unclear, confusing, slow, or difficult for real users. For example, a checkout page may technically calculate the correct total, but users may still not understand how to change product quantities or complete the checkout process. Similarly, the supplier dashboard may correctly update the database, but suppliers may not understand the layout or where to edit information.

The feedback process will be used to identify:
- what parts of the prototype work well
- what users find confusing
- where users hesitate or make mistakes
- whether the navigation is clear
- whether the checkout process is understandable
- whether accessibility options help users
- whether supplier dashboard features are understandable
- whether the technical implementation is reliable
- what improvements should be prioritised in the next version

The feedback will support future improvements to both the interface and the code.

---

### 1.2 What the feedback is referring to

The feedback is referring to both the visible user interface and the technical functionality behind it.

#### User interface areas being evaluated

The user interface feedback focuses on:
- homepage layout
- navigation menu
- supplier page
- supplier product page
- full products page
- basket and checkout
- login and register pages
- accessibility settings
- supplier dashboard
- add/update product forms
- sales summary display

This is mainly assessed by non-technical users, because they are likely to judge the system based on clarity, ease of use, layout and confidence.

#### Technical areas being evaluated

The technical feedback focuses on:
- system flow
- backend logic
- database handling
- login/register reliability
- validation
- checkout logic
- basket quantity handling
- accessibility setting implementation
- supplier dashboard logic
- product adding/updating
- active checkbox/product visibility
- sales summary calculations
- scalability

This is mainly assessed by technical users, because they can give feedback on whether the system is logically structured and reliable.

---

### 1.3 Main aims of feedback

The main aims of the feedback are:

1. To find out whether users can understand the purpose of the website.
2. To find out whether users can move between pages without help.
3. To test whether suppliers and products are easy to find.
4. To check whether the checkout process is clear.
5. To check whether changing product quantities in the basket is understandable.
6. To check whether login/register feels familiar and usable.
7. To test whether accessibility settings are useful.
8. To find out whether the supplier dashboard is understandable.
9. To check whether supplier details and products can be updated logically.
10. To find out whether technical users think the backend logic is suitable.
11. To identify repeated problems across surveys, observation and interviews.
12. To decide which improvements should be made first.

---

### 1.4 Why different feedback methods are needed

One feedback method alone would not be reliable enough. For example, a user may say in a survey that the product page is easy to use, but observation may show that they hesitated or clicked the wrong area before finding the right product. An interview can then explain why they struggled.

Because of this, three main feedback methods will be used:

1. **Surveys**
   - Used to collect rating scores and written opinions.

2. **Task-based observation**
   - Used to record what users actually do when using the prototype.

3. **Semi-structured interviews**
   - Used to understand why users experienced certain issues.

Using these methods together allows triangulation, which makes the feedback more reliable.

---

### 1.5 Link to future improvements

The feedback will be used to guide improvements in the next version of the prototype.

Possible improvements may include:
- improving navigation labels
- making buttons clearer
- improving the checkout layout
- adding stronger confirmation messages
- improving validation
- adding better error messages
- simplifying the supplier dashboard
- improving accessibility settings
- improving the product management process
- making the sales summary easier to understand
- improving database logic
- improving mobile responsiveness
- improving overall performance

The final improvement decisions will be based on evidence from survey scores, observation notes and interview responses.

## Document 2 — Target User Document

### 2.1 Purpose of identifying target users

Target users need to be identified so that feedback is collected from suitable people. Different user groups will notice different issues. Non-technical users are useful for judging usability, layout and navigation, while technical users are useful for judging backend logic, validation, database handling and scalability.

Using both groups gives a more balanced evaluation of the GLH prototype.

---

### 2.2 Non-technical users

Non-technical users represent normal users who may use the website as customers or suppliers without programming knowledge.

These users are expected to focus on:
- ease of use
- first impressions
- navigation
- layout clarity
- button wording
- product browsing
- checkout clarity
- accessibility usefulness
- supplier dashboard understandability

Non-technical users are important because the system must be understandable to people who do not know how the code works. If the system is technically functional but users do not understand how to use it, then the prototype still needs improvement.

---

### 2.3 Technical users

Technical users represent people with some knowledge of programming, databases, web systems or software logic.

These users are expected to focus on:
- system structure
- database implementation
- login/register reliability
- validation
- checkout logic
- product management logic
- accessibility implementation
- dashboard functionality
- active checkbox logic
- sales summary usefulness
- scalability
- reliability

Technical users are important because they can identify weaknesses that normal users may not notice. For example, a normal user may think the login page looks fine, while a technical user may identify weak validation or insecure password handling.

---

### 2.4 Why both groups are needed

Both groups are needed because they provide different types of insight.

Non-technical users answer questions such as:
- Is the system easy to use?
- Is the layout clear?
- Can users find products?
- Is checkout understandable?
- Do accessibility settings help?

Technical users answer questions such as:
- Is the database suitable?
- Is the logic reliable?
- Is validation strong enough?
- Would the system scale?
- Is the checkout implementation appropriate?

Together, this creates a more complete evaluation.

---

### 2.5 Reducing bias

Bias will be reduced by:
- using the same task list where possible
- asking structured questions
- including both positive and negative questions
- not leading users toward certain answers
- giving users the chance to explain their views
- comparing survey data with observation and interview data
- anonymising responses so users feel comfortable giving honest feedback

This improves the reliability of the feedback.
